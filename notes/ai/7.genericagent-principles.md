# GenericAgent 原理笔记

## 一句话理解

`GenericAgent` 的核心不是把上下文做长，而是把 `context density` 做高。

也就是：

- 不是“让模型看到更多”
- 而是“让模型只看到当前决策最需要的信息”

如果把这个系统浓缩成一句更直白的话，就是：

> GenericAgent 不是追求一次任务里想得更多，而是追求把做过的事情沉淀下来，让以后越来越少需要重新想。

---

## 1. 它到底在解决什么问题

现在很多 Agent 系统看起来问题很多，根源往往不是模型本身，而是上下文组织方式不对。

随着任务运行，系统里会不断堆积：

- 历史对话
- 工具定义
- 工具返回结果
- 网页文本
- 文件内容
- 执行日志
- 各种“记忆”

表面看，这像是在给模型提供更多信息；但实际结果常常相反：信息越多，模型越难聚焦，关键线索反而被淹没。

所以 `GenericAgent` 的出发点不是“继续扩上下文”，而是重新问一个更关键的问题：

> 对模型当前这一轮决策，什么信息才是真的必要的？

这就是它所谓的 `Context Engineering`。

---

## 2. 为什么长上下文不等于强 Agent

`GenericAgent` 的第一性原理，其实是对“长上下文神话”的反思。

直觉上，我们很容易认为：

- 上下文窗口越大越好
- 历史保留越完整越好
- 记住的东西越多越好

但实际并不是这样。

### 2.1 Lost-in-the-Middle

模型通常更关注上下文开头和结尾，中间内容更容易被忽略。这意味着：

- 信息不是“放进去就等于被用到”
- 很多看起来保留了的内容，其实对决策没有真正起作用

### 2.2 注意力稀释

上下文里无关信息越多，模型越难抓住重点。

这不是简单的 token 浪费问题，而是认知干扰问题。低价值信息会把高价值信息埋掉。

### 2.3 标称窗口不等于有效窗口

即使模型号称支持超长上下文，也不代表它能稳定、高质量地利用这些信息。

`GenericAgent` 引用了一个很重要的认识：真正“无明显幻觉地有效利用”的上下文长度，通常远小于标称窗口。

所以核心问题不是：

> 模型最多能看多少？

而是：

> 当前这轮到底该让模型看什么？

---

## 3. GenericAgent 的总设计观

`GenericAgent` 是一个 `model-agnostic` 的系统。

意思是：

- 底层可以接不同模型，比如 `Claude`、`GPT`、`Gemini`
- 但上层执行框架不依赖某个特定模型

也就是说，它的重点不是“押注哪个模型最强”，而是“无论模型是谁，都尽量把喂给它的上下文组织好”。

它的基本运行方式就是一个统一循环：

1. 构建当前执行上下文
2. 调用 LLM 做推理和决策
3. 调工具执行动作
4. 把结果结构化反馈回来
5. 进入下一轮

这个循环看起来普通，但关键不在循环本身，而在“上下文如何被构建”。

在 `GenericAgent` 里，执行上下文不是单纯的用户输入，而是由下面这些部分组成：

- 系统提示词
- 始终在线的核心记忆
- 工具定义
- 任务规格
- 压缩后的历史对话
- 当前工作记忆

所以它真正优化的对象，不是一段 prompt，而是“每轮模型所看到的全部决策环境”。

---

## 4. 核心原则：完备性 vs 简洁性

`GenericAgent` 对上下文设计有一个很清晰的张力定义：

- `完备性`
- `简洁性`

### 4.1 完备性

如果当前决策需要的信息没有显式出现，模型就只能猜。

所以，关键约束、任务目标、必要上下文必须明确存在。

### 4.2 简洁性

如果上下文里混入太多无关、重复、过时信息，模型即使“看到了”，也会更难聚焦。

所以，不必要的内容必须被清理掉。

### 4.3 这两者为什么永远矛盾

为了保证完备性，人会倾向于“多放一点总没坏处”。

但多放一点，很容易破坏简洁性。

而为了保证简洁性，人又会倾向于总结、抽象、压缩。

但压缩又可能丢掉关键细节，伤害完备性。

`GenericAgent` 的整套机制，本质上就是在这两个目标之间找更优平衡点。

---

## 5. 四大机制总览

围绕“提高上下文信息密度”这个目标，`GenericAgent` 做了四件事：

1. 最小原子工具集
2. 分层记忆
3. 运行时上下文截断与压缩
4. 自我进化

可以先粗略理解为：

- 工具层：解决“怎么低成本行动”
- 记忆层：解决“怎么长期积累但不污染当前上下文”
- 运行时层：解决“单任务里上下文怎么防膨胀”
- 演化层：解决“做过一次的事以后怎么变便宜”

下面逐个记。

---

## 6. 最小原子工具集：不是工具越多越强

这一点是 `GenericAgent` 很反直觉、但又很重要的地方。

很多 Agent 框架的思路是：

- 新需求出现
- 加一个新工具
- 再出现新场景
- 再加一个新工具

结果就是工具越来越多，prompt 也越来越重。

### 6.1 为什么工具多会有问题

工具不是“免费能力”，而是固定上下文成本。

每个工具都会带来：

- 描述文本
- 参数说明
- 使用边界
- 调用选择空间

这会同时带来两个负担：

- `prompt level`：每轮都要注入工具定义，占上下文
- `policy level`：模型面对的动作空间变大，更难选对工具

也就是说，工具一多，不只是 token 多了，决策歧义也变大了。

### 6.2 GenericAgent 的做法

`GenericAgent` 追求的是 `最小原子工具集`。

设计哲学不是“瑞士军刀”，而是“乐高积木”：

- 每个工具只提供一种不可再分的原始能力
- 复杂任务通过工具组合完成
- 不轻易为了一个场景发明一个专用工具

它的原子工具覆盖的大类包括：

- 文件操作
- 代码执行
- 网页交互
- 记忆管理
- 人机协作

### 6.3 它具体怎么做

可以把它理解成三步：

1. 先只保留不可替代的基础能力
2. 再把复杂任务拆成这些基础能力的组合
3. 尽量不为了单一场景新增专用工具

落到执行层，大概就是：

- 读文件，不提供一个“万能项目分析器”，而是提供基础读取能力
- 改文件，不提供很多花哨写法，而是提供明确、可控的修改能力
- 看网页，不把整页原始内容全塞进上下文，而是先做结构化提取
- 跑代码，用一个统一执行入口处理大量外部动作

所以它不是靠“预制大量高级工具”来增强能力，而是靠“用少量稳定工具覆盖最多真实任务”来降成本。

### 6.4 一个关键理解

`GenericAgent` 其实承认一个底层事实：

> 理论上，只靠 `code_run`，Agent 就可以写脚本去模拟很多其他工具能力。

所以其他工具存在的意义，不是扩展理论能力上限，而是：

- 降低高频动作成本
- 减少上下文注入负担
- 减少调用歧义

这很重要，因为它说明 GA 的工具设计目标从来不是“无限加能力”，而是“保留必要能力，同时尽量瘦身”。

## 7. 分层记忆：不是把历史都塞进去

很多系统处理记忆的方式都很粗暴：

- 把更多历史放进 prompt
- 希望模型自己“记住”

`GenericAgent` 认为这不是真正的记忆，只是“历史追加”。

因为一旦所有历史都顺着堆进去，结果往往是：

- 历史看起来保留了
- 但关键经验并没有被提纯
- 当前上下文反而被大量旧信息污染

### 7.1 它的核心思路

`GenericAgent` 的记忆理念可以概括成一句话：

> 知道信息在哪里，但不提前加载。

也就是：

- 记忆应该存在系统外部
- 当前 prompt 默认只带最小导航信息
- 真需要某类知识时，再按需读取

### 7.2 它具体怎么做

这里可以按一个很清楚的流程理解：

1. 先把长期知识放到上下文外部
2. 在上下文里只保留最小索引
3. 当前任务需要时，再读取对应事实或 SOP
4. 执行完成后，再决定哪些内容值得写回长期记忆

这样做的直接效果是：

- 启动时上下文更轻
- 不会把所有旧经验都提前注入
- 当前任务只会接触到和自己相关的知识
- 长期知识库可以增长，但活跃 prompt 不会线性膨胀

### 7.3 四层记忆架构

它把记忆分成四层：

#### L1：索引层

只保存“有什么知识存在、去哪里找”的路由信息。

特点：

- 始终在线
- 必须有界
- 不存正文

#### L2：事实层

存经过验证、可跨任务复用的稳定事实。

比如：

- 某个项目的结构规律
- 某类环境配置的稳定结论
- 某些长期成立的事实知识

#### L3：SOP 层

存已经被验证过的可复用流程。

比如：

- 某类排错怎么做
- 某类调研任务怎么拆
- 某类仓库初始化流程怎么执行

#### L4：归档层

保存压缩后的历史记录。

特点是：

- 归档存在
- 默认不进入活跃上下文

### 7.4 这套设计真正重要的地方

重点不是“分成了四层”，而是：

- 默认只加载最少内容
- 正文按需读取
- 长期知识要经过筛选，不是历史自然沉淀

### 7.5 No Execution, No Memory

这是 `GenericAgent` 记忆设计里最重要的一条原则之一：

> 没有执行验证过的内容，不进入长期记忆。

也就是：

- 不是模型觉得有道理就能记
- 不是总结得像经验就能记
- 必须经过实际执行或验证

它背后的写回逻辑可以简单记成：

1. 任务执行
2. 结果验证
3. 判断是否具备跨任务复用价值
4. 有价值的进入 `L2` 或 `L3`
5. 原始历史进入归档，而不是直接变成长期知识

这个规则的作用，是防止记忆被模型臆断、错误总结和低质量抽象污染。

### 7.6 元记忆和工作记忆

除了长期记忆，GA 还有两个辅助概念。

#### 元记忆

元记忆不存业务知识，而是存“记忆系统的规则”。

比如：

- 哪些层各自干什么
- 什么内容允许写进去
- 什么时候该更新

作用是防止模型乱写、乱读、乱升级记忆。

#### 工作记忆

工作记忆服务于当前任务，不是长期知识。

它通常包含：

- 当前轮次编号
- 最近若干轮的压缩摘要
- 当前任务里的关键约束和关键结论

作用是维持单任务连续性，而不是沉淀长期资产。

## 8. 运行时上下文管理：单任务里也不能任其膨胀

即使已经有了分层记忆，单个任务内部依然会不断膨胀。

原因很简单：

- 每轮会增加对话
- 每轮会增加工具结果
- 每轮可能读网页、文件、日志
- 任务越长，累积越多

所以 `GenericAgent` 还需要一套“运行中的上下文治理机制”。

它不是等窗口爆了再想办法，而是边跑边清理。

### 8.1 它具体怎么做

可以把运行时管理理解成一条流水线：

1. 工具结果先裁剪
2. 旧轮次消息再压缩
3. 还是太长就驱逐最旧消息
4. 始终保留一个工作记忆锚点维持任务主线

所以它的逻辑不是“留住所有历史”，而是：

- 尽量保留对当前决策最有用的部分
- 尽量删掉已经过期、重复、低价值的部分
- 删除旧消息时，用锚点保住任务连续性

### 8.2 Stage 1：工具级截断

工具结果在进入上下文前，先被裁剪。

不是把原始结果无脑整段贴进去，而是优先保留最可能有用的部分。

这里有个很实用的经验：

- 很多关键错误和最终状态都在输出尾部
- 所以保留头尾通常比保留中间更有价值

### 8.3 Stage 2：标签级压缩

系统会周期性地压缩较旧轮次中的结构化内容，比如：

- `<thinking>`
- `<tool_use>`
- `<tool_result>`

同时还会处理旧的：

- 历史摘要
- 关键点快照

对于过期内容，不是继续完整保留，而是替换为占位符或压缩表达。

这一层的价值在于：

- 旧轮次的 token 大幅下降
- 但结构主线还在
- 还能提高 prompt cache 命中率

### 8.4 Stage 3：消息驱逐

如果压缩之后还是超预算，就继续做更激进的处理：

- 先重跑一轮更强压缩
- 再按时间顺序删除最早消息
- 同时修复必要结构，保证上下文协议合法

这里 `GenericAgent` 有个很明确的取舍：

> 不把“再调用一个 LLM 来生成摘要”当作主解决方案。

原因是：

- 摘要本身有成本
- 摘要本身也可能失真
- 摘要写出来还是要占 token

### 8.5 Stage 4：工作记忆锚点

真正保证任务不会“失忆”的，是工作记忆锚点。

每次工具调用之后，系统会维护一个锚点，里面通常有：

- 最近若干轮的单行摘要
- 当前轮次编号
- `key_info`

这个锚点的作用很像“任务主线提示板”：

- 我已经做到哪一步
- 当前任务目标是什么
- 哪些约束不能丢
- 哪些结论已经确认

这样即使旧消息被删掉很多，任务主线依然能维持。

## 9. 自我进化：不是重复完成，而是逐渐收敛

这是 `GenericAgent` 最像“长期系统”而不是“一次性 Agent”的地方。

很多 Agent 虽然能完成任务，但有一个明显问题：

- 同类任务做了很多次
- 下一次还是重新探索
- 没有明显变便宜

也就是说，它没有真正“学会”。

### 9.1 GenericAgent 的判断

GA 的核心判断是：

> 进化的应该是策略，而不是工具。

它不会一边运行一边改变底层工具体系，而是让经验沉淀成两类可进化资产：

- `SOP`
- 可复用脚本

### 9.2 它具体怎么做

这部分可以直接记成一个收敛链路：

1. 第一次做任务时，自然语言探索
2. 找到一条可行路径后，提炼成 SOP
3. 多次验证 SOP 后，把稳定部分代码化
4. 以后遇到类似任务，优先走 SOP 或脚本
5. 遇到新变化，再做小范围探索和修正

这样系统会逐渐从：

- “每次都重新想”

变成：

- “大部分执行直接复用，只有少量边缘情况才重新想”

### 9.3 三阶段演化路径

#### 第一阶段：自然语言探索执行

刚开始做一类任务时，Agent 需要自己试、自己查、自己摸索。

特点：

- LLM 调用多
- token 消耗高
- 时间长
- 不稳定

#### 第二阶段：SOP 蒸馏

当某条成功路径逐渐稳定后，把它压缩成 SOP。

这时候再遇到类似任务，Agent 不用从零想“下一步是什么”，而是：

- 先读 SOP
- 按流程执行
- 遇到偏差再局部修正

#### 第三阶段：代码化执行

当 SOP 足够稳定后，再进一步把它编译成脚本或程序化流程。

这意味着：

- 模型不再负责大量低层推理
- 只需要做调度、判断和适配
- 重复任务成本会显著下降

### 9.4 这个机制真正厉害的地方

它让系统的长期目标从：

- “每次都聪明地重新思考”

变成：

- “把需要思考的部分越来越少”

这比单次任务表现更重要。

因为一个真正可长期运行的 Agent，不该永远停留在“高成本推理”状态。

## 10. 为什么说它会“越用越强”

“越用越强”这句话很多系统都在说，但 `GenericAgent` 这里是有结构支撑的。

原因不是模型在线微调了，也不是底层自动训练了，而是：

1. 成功经验会被提炼
2. 提炼后的经验能被放进 `L2/L3`
3. 后续任务会通过 `L1` 路由到这些经验
4. 相似任务因此不再从零开始
5. 最终从探索收敛到 SOP，再收敛到脚本

所以它增长的不是“原始历史”，而是“高信息密度的可复用策略”。

这也是为什么 GA 不是单纯的记忆系统，而是一个带演化闭环的 Agent 系统。

这里可以把“越用越强”具体拆成：

- 强在不是保留了更多历史
- 强在把历史压缩成了更好用的策略
- 强在后续任务能自动路由到这些策略
- 强在重复任务的 token、时间和调用次数都能下降

## 11. 涌现能力：复杂功能不是靠堆模块得来的

`GenericAgent` 很有意思的一点是，它并没有靠庞大框架去实现那些看起来很高级的能力。

像下面这些东西：

- 子 Agent
- 看门狗
- 定时任务
- 自主空闲行为

在很多系统里，可能都会被做成复杂模块。

但在 GA 里，它们更像是少量基础原语自然组合出来的结果。

### 11.1 三个底层原语

#### 1. 自托管 CLI

Agent 本身就是一个可以被调用的命令行进程。

这意味着：

- 它可以被人启动
- 也可以被另一个 Agent 启动

#### 2. 文件协议

通过标准输入输出文件完成跨进程协作。

比如任务目录中不同文件可以承担：

- 输入
- 输出
- 回复
- 注入上下文
- 注入消息

#### 3. Reflect Mode

也就是：

- 持续轮询某个条件
- 条件满足时生成任务
- 然后走标准 Agent 流水线

### 11.2 它具体怎么做

这部分最好直接按“功能 = 原语组合”来记：

- `子 Agent` = 再启动一个新的 GA 进程
- `看门狗` = 监控脚本 + 条件触发 + 标准任务流
- `定时任务` = 时间条件 + Reflect Mode
- `自主空闲行为` = 空闲检测 + 自动投递任务

也就是说，GA 不会先设计一个庞大的“高级能力中心”，而是：

1. 先定义少量稳定原语
2. 再让复杂行为从这些原语里组合出来
3. 这样既减小框架体积，也降低维护成本

### 11.3 这些高级能力怎么涌现出来

有了上面三个原语之后：

- `子 Agent`：本质上就是再启动一个 GA 进程
- `看门狗`：本质上就是写一个监控条件的触发脚本
- `定时任务`：本质上就是“时间到了就触发”
- `自主空闲行为`：本质上就是在空闲状态下自动投递一个自我提升任务

所以它的设计哲学很明确：

> 复杂能力不一定要靠复杂模块实现，也可以靠极少数稳定原语低成本组合出来。

这其实和前面的所有设计是一致的：持续做减法，而不是堆功能。

## 12. 我对这套设计的总体理解

如果把前面所有内容收成一张脑图，`GenericAgent` 大概是在做这件事：

- 用最少的工具定义，减少固定上下文成本
- 用分层记忆，把长期知识放在上下文外部
- 用运行时压缩，控制单任务内部的膨胀
- 用 SOP 和脚本化，把成功经验变成未来低成本执行

如果要把整套运行逻辑写成一个最短闭环，可以记成：

1. 用最小工具完成任务
2. 用分层记忆控制长期知识注入
3. 用运行时压缩控制当前 token 膨胀
4. 用 SOP 和脚本把成功经验沉淀下来
5. 下次再遇到类似任务时，优先复用已有策略

所以它的重点并不是：

- 一次任务里把模型压榨到最强

而是：

- 把系统做成一个长期可运行、可积累、可收敛的执行体

换句话说，GA 真正关心的不是“单轮推理最聪明”，而是“长期运行后整体效率越来越高”。

---

## 13. 最后记一句最重要的结论

我觉得这套原理里最值得记住的一句话是：

> Agent 的关键不是拥有更多上下文，而是持续控制上下文里的噪声，并把经验压缩成以后更便宜的决策。

或者说得更短一点：

> `GenericAgent = 最小工具 + 分层记忆 + 运行时压缩 + SOP 演化`

再短一点：

> `不是让 Agent 一次想更多，而是让它以后越来越少重新想。`
