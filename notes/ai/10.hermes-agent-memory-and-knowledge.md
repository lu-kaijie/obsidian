# 记忆与知识系统 — 实现解析

## 概述

Hermes Agent 的记忆与知识系统使 Agent 能够跨会话保持对用户和环境的认知。
系统分为三个层次：**持久化记忆**（文件存储）、**会话搜索**（FTS5 全文检索）、
**外部记忆提供者**（插件化后端）。三者协同工作，让 Agent 既能记住关键事实，
又能回溯历史对话，还能通过第三方服务实现深度用户建模。

核心源码文件：

| 文件 | 职责 |
|------|------|
| `tools/memory_tool.py` | 内置记忆工具（MEMORY.md + USER.md） |
| `agent/memory_manager.py` | 记忆提供者编排器 |
| `agent/memory_provider.py` | 插件化记忆后端 ABC |
| `hermes_state.py` | SQLite 会话存储 + FTS5 索引 |
| `tools/session_search_tool.py` | 会话搜索 + LLM 摘要 |
| `run_agent.py` | Memory Nudge 触发 + 后台审查 |

## 1. 内置记忆工具（memory_tool.py）

### 1.1 双存储设计

系统维护两个独立的记忆文件：

| 文件 | 用途 | 示例内容 |
|------|------|----------|
| `~/.hermes/memories/MEMORY.md` | Agent 的个人笔记 | 环境事实、项目约定、工具技巧 |
| `~/.hermes/memories/USER.md` | 关于用户的认知 | 偏好、沟通风格、工作习惯 |

条目以 `§`（section sign）分隔，支持多行：

```markdown
§
用户偏好简洁的代码注释，不喜欢冗长的解释
§
项目使用 uv 而非 pip 管理依赖，虚拟环境在 .venv/
§
用户的时区是 UTC+8，工作时间通常是 10:00-22:00
```

### 1.2 冻结快照模式

**关键设计决策**：记忆在会话开始时作为冻结快照注入系统提示。
会话中途的写入立即持久化到磁盘，但**不修改当前系统提示**。

这保证了：
- 整个会话的前缀缓存（prefix cache）保持有效
- 避免了中途修改上下文导致的缓存失效（成本显著增加）
- 下次会话自动获取最新记忆

### 1.3 工具操作

`memory` 工具支持四种操作：

```python
# action 参数
"add"     # 添加新条目
"replace" # 用子串匹配定位并替换
"remove"  # 用子串匹配定位并删除
"read"    # 读取当前记忆内容
```

`replace` 和 `remove` 使用**短唯一子串匹配**，不需要完整文本或 ID。

### 1.4 安全扫描

写入记忆前会进行威胁模式扫描，检测：
- 提示注入（"ignore previous instructions"）
- 角色劫持（"you are now"）
- 凭证外泄（curl/wget + API_KEY）
- SSH 后门（authorized_keys）

被标记的内容会被拒绝写入，防止恶意内容通过记忆注入系统提示。

## 2. Memory Nudge 机制

### 2.1 触发逻辑

与技能 nudge 类似，记忆也有自己的迭代计数器：

```python
# run_agent.py AIAgent.__init__
self._memory_nudge_interval = 10  # 默认每 10 次迭代
mem_config = config.get("memory", {})
self._memory_nudge_interval = int(mem_config.get("nudge_interval", 10))
```

### 2.2 审查 Prompt

`_MEMORY_REVIEW_PROMPT` 指导后台 agent 关注：

1. 用户是否透露了关于自己的信息（人格、偏好、个人细节）
2. 用户是否表达了对 agent 行为方式的期望

如果有值得保存的内容，调用 memory 工具写入；否则直接停止。

### 2.3 与技能审查的协同

当两个 nudge 同时触发时，使用 `_COMBINED_REVIEW_PROMPT` 合并执行，
避免 fork 两个独立的审查 agent，节省模型调用成本。

## 3. 会话搜索系统

### 3.1 SQLite FTS5 索引

`hermes_state.py` 中定义了两个全文搜索虚拟表：

**标准 FTS5 表**（unicode61 分词器，适合英文）：
```sql
CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts USING fts5(content);

-- 自动同步触发器
CREATE TRIGGER messages_fts_insert AFTER INSERT ON messages BEGIN
    INSERT INTO messages_fts(rowid, content) VALUES (
        new.id,
        COALESCE(new.content, '') || ' ' ||
        COALESCE(new.tool_name, '') || ' ' ||
        COALESCE(new.tool_calls, '')
    );
END;
```

**Trigram FTS5 表**（三字节序列分词，支持 CJK）：
```sql
CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts_trigram USING fts5(
    content,
    tokenize='trigram'
);
```

Trigram 索引解决了 unicode61 分词器将 CJK 字符拆分为单字 token 的问题，
通过重叠的 3 字节序列实现任意脚本的子串匹配。

### 3.2 session_search 工具流程

`tools/session_search_tool.py` 实现了完整的搜索-摘要管线：

```
用户查询
   │
   ▼
FTS5 搜索（按相关性排序）
   │
   ▼
按 session 分组，取 Top N（默认 3）
   │
   ▼
加载每个 session 的对话（截断到 ~100k 字符，以匹配点为中心）
   │
   ▼
发送到辅助模型（auxiliary.session_search）进行摘要
   │
   ▼
返回结构化的 per-session 摘要 + 元数据
```

关键参数：
- `MAX_SESSION_CHARS = 100_000` — 每个 session 的最大上下文
- `MAX_SUMMARY_TOKENS = 10000` — 摘要的最大 token 数
- `max_concurrency`（默认 3，最大 5）— 并发摘要请求数

### 3.3 辅助模型路由

搜索摘要使用 `auxiliary.session_search` 配置的模型，默认回退到主聊天模型。
用户可以为此任务指定更便宜/更快的模型，不影响主对话质量。

## 4. 外部记忆提供者（插件系统）

### 4.1 MemoryProvider ABC

`agent/memory_provider.py` 定义了插件化记忆后端的抽象接口：

```python
class MemoryProvider(ABC):
    @property
    def name(self) -> str: ...          # 标识符（honcho/mem0/hindsight）

    def is_available(self) -> bool: ... # 检查配置和依赖
    def initialize(self, session_id, **kwargs): ...  # 会话初始化
    def system_prompt_block(self) -> str: ...        # 系统提示注入
    def prefetch(self, query) -> str: ...            # 预取相关记忆
    def sync_turn(self, user_msg, asst_msg): ...    # 每轮同步
    def get_tool_schemas(self) -> list: ...          # 暴露工具
    def handle_tool_call(self, name, args): ...     # 处理工具调用
    def shutdown(self): ...                          # 清理退出
```

可选钩子：
- `on_session_end(messages)` — 会话结束时提取
- `on_pre_compress(messages)` — 上下文压缩前提取
- `on_memory_write(action, target, content)` — 镜像内置记忆写入
- `on_delegation(task, result)` — 观察子代理工作

### 4.2 MemoryManager 编排

`agent/memory_manager.py` 强制**单外部提供者**限制：

```python
class MemoryManager:
    def add_provider(self, provider): ...
    def build_system_prompt(self) -> str: ...
    def prefetch_all(self, query) -> str: ...
    def sync_all(self, user_msg, asst_msg): ...
```

设计约束：同一时间只允许一个外部提供者运行，防止工具 schema 膨胀和后端冲突。

### 4.3 已有提供者

内置支持的外部记忆后端（`plugins/memory/` 下）：

- **Honcho** — 辩证法用户建模（dialectic user modeling）
- **Mem0** — 结构化记忆图谱
- **Hindsight** — 回顾式记忆提取
- **Supermemory** — 语义记忆搜索
- **Holographic** — 全息记忆编码

通过 `memory.provider` 配置项激活，`hermes memory setup` 交互式配置。

## 5. 上下文安全

### 5.1 StreamingContextScrubber

`memory_manager.py` 中的流式清洗器防止记忆上下文泄露到用户可见输出：

```python
class StreamingContextScrubber:
    """跨 chunk 边界的状态机，过滤 <memory-context> 标签"""
    def feed(self, delta) -> str: ...  # 返回可见部分
    def flush(self) -> str: ...        # 流结束时刷新
```

### 5.2 sanitize_context

对提供者返回的内容进行清洗：
- 移除 `<memory-context>...</memory-context>` 块
- 移除 `[System note: ...]` 注入标记
- 移除裸露的 fence 标签

## 6. 完整架构图

```
┌─────────────────────────────────────────────────────┐
│                   系统提示                            │
│  ┌──────────────┐  ┌──────────────┐                 │
│  │ MEMORY.md    │  │ USER.md      │  ← 冻结快照     │
│  │ (agent笔记)  │  │ (用户画像)   │                 │
│  └──────────────┘  └──────────────┘                 │
│  ┌──────────────────────────────────┐               │
│  │ 外部提供者 system_prompt_block() │  ← Honcho等   │
│  └──────────────────────────────────┘               │
└─────────────────────────────────────────────────────┘
         │                        │
         ▼                        ▼
┌─────────────────┐    ┌─────────────────────┐
│  memory 工具     │    │  session_search 工具 │
│  add/replace/   │    │  FTS5 → 分组 → 摘要  │
│  remove/read    │    │                     │
└────────┬────────┘    └──────────┬──────────┘
         │                        │
         ▼                        ▼
┌─────────────────┐    ┌─────────────────────┐
│ ~/.hermes/      │    │ state.db            │
│ memories/       │    │ messages_fts        │
│ MEMORY.md       │    │ messages_fts_trigram │
│ USER.md         │    │ (SQLite + WAL)      │
└─────────────────┘    └─────────────────────┘
         │
         ▼
┌─────────────────────────────────────┐
│  Memory Nudge（每 10 次迭代）        │
│  → Fork 后台 Agent                  │
│  → 分析对话 → 写入记忆              │
└─────────────────────────────────────┘
```

这三层系统协同工作：内置记忆提供快速的事实召回，会话搜索提供
深度的历史回溯，外部提供者实现高级的用户建模和语义理解。
