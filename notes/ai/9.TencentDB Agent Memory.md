# TencentDB Agent Memory 项目做法总结

## 1. 这个项目在解决什么问题

这个项目不是一个普通的“给 Agent 接一个向量库”的插件，而是在解决两个更具体的问题：

1. 长任务里，上下文会被工具输出、报错日志、搜索结果快速撑爆。
2. 跨会话里，Agent 很难稳定记住用户偏好、项目背景、长期约束和历史决策。

它的核心判断是：

- 不能把所有历史都一直塞进上下文。
- 也不能简单粗暴地把历史压成一段摘要，因为这样会丢失可追溯性。
- 记忆系统必须“分层”，并且高层结构必须能回钻到低层原始证据。

所以它采用了两条主线同时推进：

- 长期记忆：`L0 -> L1 -> L2 -> L3` 的四层记忆流水线。
- 短期记忆：把当前任务中超长的上下文卸载到外部，只在 prompt 里保留一个高密度符号化表示。

## 2. 总体设计思路

项目的总体做法可以概括成一句话：

**底层保留原始证据，上层提炼结构化认知，召回时优先给结构，不够再回钻细节。**

对应到实现上，分成两套系统：

### 2.1 长期记忆系统

长期记忆是这个项目的主架构，核心在四层抽象：

- `L0 Conversation`：原始对话。
- `L1 Atom`：从对话中抽取出的结构化记忆原子。
- `L2 Scenario`：把零散记忆组织成场景块。
- `L3 Persona`：把多个场景再汇总成用户画像和长期偏好。

这条链路的意义是：

- `L0` 保存证据。
- `L1` 保存可检索事实。
- `L2` 保存上下文化的故事和阶段。
- `L3` 保存稳定偏好、风格、工作方式。




  这个项目的长期记忆，可以理解成“不是把聊天记录一股脑存起来”，而是像人整理资料一样，分 4 层慢慢加工。

  - L0 Conversation：最原始的聊天记录。
    就像会议录音或聊天截图，内容最完整，但也最杂，适合留作原始依据。
  - L1 Atom：从原始对话里提炼出的“单条有价值信息”。
    比如“用户偏好中文回答”“用户在做 Agent Runtime 项目”“用户希望回答简洁直接”。
    它像一张张信息卡片，方便后面搜索。
  - L2 Scenario：把很多零散信息卡片，按主题整理成一个“场景档案”。
    比如把和“Agent 架构设计”相关的历史讨论放进一个场景，把和“面试准备”相关的内容放进另一个场景。
    这样系统记住的就不只是碎片，而是一段有上下文的经历。
  - L3 Persona：再把多个场景继续往上总结，形成对这个用户的整体认识。
    比如这个用户通常关注什么方向、喜欢什么表达方式、做决策时重视什么。
    这一层更像“长期画像”。

  这 4 层的关系，可以类比成：

  - L0 像原始聊天记录
  - L1 像从聊天里摘出来的重点笔记
  - L2 像按主题整理好的文件夹
  - L3 像对这个人的整体印象和总结

  这样设计的好处是：

  - L0 保留原始依据，防止信息失真
  - L1 方便检索具体事实
  - L2 方便理解事情的来龙去脉
  - L3 方便跨会话保持对用户的稳定理解

### 2.2 短期上下文压缩系统

短期系统主要处理“当前这次任务太长”的问题。

它的策略不是继续堆 tokens，而是：

- 把完整工具输出和长日志落到外部文件。
- 从这些中间结果提取节点、关系、状态变化。
- 用 Mermaid 图这种高密度符号格式表达任务进度。
- 在 prompt 中只注入精简后的 Mermaid 结构和少量必要摘要。
- 当模型需要细节时，再靠 `node_id` 去追索原文。

这套设计的目标不是“摘要得更像人”，而是“让模型以更低 token 成本保留任务拓扑和可恢复性”。

## 3. 代码层面的核心架构

### 3.1 入口层：宿主适配而不是业务堆积

入口文件是 [index.ts](/home/mi/TencentDB-Agent-Memory/index.ts)。

这里的做法很克制，主要负责几件事：

- 解析配置。
- 注册 OpenClaw hooks 和工具。
- 初始化数据目录、存储和上报器。
- 把宿主事件翻译成统一的 `TdaiCore` 调用。
- 在配置开启时注册 offload 模块。

也就是说，`index.ts` 本身并不承载真正的记忆逻辑，而是一个薄壳。

### 3.2 核心编排层：统一收口到 `TdaiCore`

核心入口在 [src/core/tdai-core.ts](/home/mi/TencentDB-Agent-Memory/src/core/tdai-core.ts)。

这个类的作用是把记忆能力抽象成宿主无关的统一接口：

- `handleBeforeRecall`：在模型推理前召回记忆。
- `handleTurnCommitted`：在一轮对话完成后写入 L0 并触发流水线。
- `handleMemorySearch` / `handleConversationSearch`：给 Agent 提供主动查询记忆的工具。
- `destroy`：关闭调度器、存储和后台任务。

这一步很关键，因为它把“记忆能力”和“OpenClaw/Hermes 运行时”拆开了。

结果是：

- OpenClaw 场景下可以走插件模式。
- Hermes 场景下可以走独立 Gateway。
- 后续如果接别的 Agent 宿主，也不必重写整套记忆逻辑。

### 3.3 宿主适配层：OpenClaw 与 Gateway 双运行时

项目并不是只为 OpenClaw 写的。

- OpenClaw 侧通过 `OpenClawHostAdapter` 接入。
- Hermes 侧通过 [src/gateway/server.ts](/home/mi/TencentDB-Agent-Memory/src/gateway/server.ts) 暴露 HTTP 接口。

Gateway 提供的接口也很清晰：

- `/recall`
- `/capture`
- `/search/memories`
- `/search/conversations`
- `/session/end`
- `/seed`

这说明作者把“记忆系统”当成一个可复用的 sidecar 服务，而不是写死在某个 Agent 框架里。

## 4. 长期记忆流水线是怎么做的

### 4.1 L0：先完整记录，再决定怎么提炼

L0 记录逻辑在 [src/core/hooks/auto-capture.ts](/home/mi/TencentDB-Agent-Memory/src/core/hooks/auto-capture.ts) 和 [src/core/conversation/l0-recorder.ts](/home/mi/TencentDB-Agent-Memory/src/core/conversation/l0-recorder.ts)。

这里的核心原则是：

- 每轮结束先把原始对话可靠落地。
- 记录和 checkpoint 更新用原子方式做，避免并发重复采集。
- 先保证证据完整，再考虑后续抽取。

这个阶段还会做一件很实用的事：

- 如果底层存储支持延迟 embedding，就先写元数据和全文索引，embedding 放后台做。

这意味着它优先保证主流程不被 embedding 调用卡住。

### 4.2 L1：把对话抽成“记忆原子”

L1 提取逻辑在 [src/core/record/l1-extractor.ts](/home/mi/TencentDB-Agent-Memory/src/core/record/l1-extractor.ts)。

做法不是“每条消息直接入库”，而是：

1. 对 L0 消息做质量过滤。
2. 划分新消息和背景消息。
3. 让 LLM 一次完成“场景分段 + 记忆提取”。
4. 对提取出的记忆做冲突检测和去重。
5. 再写成 L1 结构化记录。

这背后的判断是：

- 原始消息不适合作为长期记忆单元。
- 记忆应该是结构化事实、偏好、约束、事件，而不是聊天碎片。

### 4.3 L2：把零散记忆继续组织成场景块

L2 在 [src/core/scene/scene-extractor.ts](/home/mi/TencentDB-Agent-Memory/src/core/scene/scene-extractor.ts)。

这个层级很有特点，它不是把所有 L1 再压成一份总表，而是做“场景化组织”：

- 先读取已有 scene index。
- 把新记忆和已有场景摘要一起交给 LLM。
- 让 LLM 在受限工作区里读写 `scene_blocks/*.md`。
- 控制场景数上限，必要时要求 merge。
- 再同步 scene index 和导航结构。

这里的设计重点是“让 LLM 直接维护结构化 Markdown 场景文件”，而不是只返回 JSON 让代码组装。

好处有两个：

- 人可以直接阅读和审查场景文件。
- 场景本身已经是可注入、可导航、可再加工的知识载体。

### 4.4 L3：从场景再归纳出 Persona

L3 在 [src/core/persona/persona-generator.ts](/home/mi/TencentDB-Agent-Memory/src/core/persona/persona-generator.ts)。

它不是每轮都全量重写 Persona，而是：

- 读取现有 Persona。
- 检查哪些 scene 自上次 Persona 之后发生变化。
- 把“变更场景”重点交给 LLM。
- 生成或增量更新 `persona.md`。
- 再附上新的 scene navigation。

这说明作者不想把 Persona 当成一次性摘要，而是把它当成一个持续演化的高层用户模型。

## 5. 流水线调度为什么这样设计

调度器在 [src/utils/pipeline-manager.ts](/home/mi/TencentDB-Agent-Memory/src/utils/pipeline-manager.ts)。

这部分的做法比“每轮都抽取”成熟很多，核心有四点。

### 5.1 不同步重处理，改成异步分层调度

每轮对话结束时只做轻量 capture，不立刻跑完整 L1/L2/L3。

原因很现实：

- LLM 抽取和 embedding 成本高。
- 每轮都跑全链路会拖慢主任务。
- 记忆生成是后台能力，不应该卡用户当前回答。

### 5.2 L1 用“阈值 + 空闲触发”

L1 的触发策略不是固定定时器，而是：

- 对话轮数达到阈值时立即触发。
- 如果对话不够多，但会话空闲一段时间，也触发。

这样兼顾了两类场景：

- 高频连续对话。
- 稀疏但重要的短会话。

### 5.3 新会话用 warm-up

项目给新 session 设计了 warm-up 阈值：`1 -> 2 -> 4 -> ... -> everyN`。

这个设计很聪明，因为：

- 新会话前几轮信息密度通常最高。
- 早期尽快提取能更快形成可用记忆。
- 会话成熟后再降低频率，减少成本。

### 5.4 L2 用 downward-only timer

L2 不是简单 cron，而是一个只能提前、不能延后的定时器模型。

本质上它在同时满足两个要求：

- 新记忆出现后，希望尽快组织成场景。
- 但又不能过于频繁地反复跑场景整理。

这让 L2 既有响应性，也有节制。

## 6. 召回是怎么做的

召回逻辑在 [src/core/hooks/auto-recall.ts](/home/mi/TencentDB-Agent-Memory/src/core/hooks/auto-recall.ts)。

它不是简单“搜一下向量库然后把结果拼进去”，而是三层联合注入：

- L1：相关记忆片段，放到 `prependContext`。
- L3：用户 Persona，放到稳定的系统上下文。
- L2：场景导航，告诉模型应该去哪看更完整的上下文。

其中有两个很重要的实现判断。

### 6.1 把动态和稳定上下文拆开

项目把 recall 结果拆成两类：

- 动态部分：本轮相关 L1 记忆。
- 稳定部分：Persona、Scene Navigation、工具调用指南。

这样做的目的是兼顾 prompt caching：

- 动态内容每轮变。
- 稳定内容尽量长期不变，便于缓存命中。

这属于很实战的 prompt 工程优化。

### 6.2 不假设“注入的内容一定够”，而是显式给出继续检索路径

项目还会给模型注入记忆工具使用指南，允许模型继续调用：

- `tdai_memory_search`
- `tdai_conversation_search`

所以它不是“召回一次定生死”，而是“先给高相关信息，再给主动深挖手段”。

## 7. 存储层为什么是异构的

存储工厂在 [src/core/store/factory.ts](/home/mi/TencentDB-Agent-Memory/src/core/store/factory.ts)。

这里体现出项目的另一个核心观点：**不同层级的记忆，不该强行放进一种介质里。**

### 7.1 下层数据放数据库

`L0/L1` 这类高频、可检索、证据导向的数据，放在数据库更合适：

- 本地默认是 SQLite。
- 支持 `sqlite-vec` 做向量检索。
- 支持 FTS5 和 BM25 做关键词检索。
- 也支持切换到 Tencent Cloud VectorDB。

SQLite 实现在 [src/core/store/sqlite.ts](/home/mi/TencentDB-Agent-Memory/src/core/store/sqlite.ts)。

它同时维护：

- `L1` 结构化记忆表 + 向量表。
- `L0` 原始对话表 + 向量表。

这意味着系统既能搜“记忆事实”，也能搜“原话证据”。

### 7.2 上层结构放 Markdown 文件

`L2/L3` 这类结构化、可读、可白盒调整的数据，放到文件系统：

- `scene_blocks/*.md`
- `persona.md`

这样做的价值非常直接：

- 人能看。
- 人能改。
- LLM 能直接读写。
- 调试和备份简单。

这比把一切都塞数据库更适合“长期演化的知识结构”。

## 8. 短期上下文卸载模块是怎么做的

这部分入口在 [src/offload/index.ts](/home/mi/TencentDB-Agent-Memory/src/offload/index.ts)。

它本质上是插件里的第二条独立子系统，专门解决单次长任务的上下文压力。

大致流程是：

1. 监听工具调用前后和 prompt 构建阶段。
2. 把大段历史、工具输出、上下文快照持久化到外部存储。
3. 通过 L1/L1.5/L2 等多级压缩步骤提取任务节点、关系和状态变化。
4. 生成 Mermaid 形式的 MMD 画布。
5. 在 prompt 中按 token 预算注入最小必要的任务结构。
6. 若发生 token overflow，再做更激进压缩。

这个模块的关键价值不是“做摘要”，而是：

- 保留任务图谱。
- 保留追溯路径。
- 把模型注意力从海量原文转移到关键状态机。

所以它更像“任务状态外存 + 符号化工作记忆”。

## 9. 为什么说这个项目不是传统 RAG

如果把它粗略理解成“带向量库的 Agent 记忆”，会低估这个项目。

它和传统 RAG 的差别主要在这里：

- 传统 RAG 偏向把文本切片后做扁平检索。
- 这个项目强调分层生成、分层存储、分层召回。

传统做法常见的问题是：

- 检索命中碎片化。
- 缺少宏观组织。
- 人很难审计记忆到底长成了什么。
- 高层摘要和底层证据之间缺乏稳定映射。

这个项目的应对方式是：

- 用 `L0/L1/L2/L3` 建立层级结构。
- 用 scene block 和 persona 作为高层白盒结构。
- 用 conversation search 和 memory search 作为回钻工具。
- 用 offload + Mermaid 解决单任务上下文膨胀。

## 10. 这个项目做法的优点

### 10.1 工程上比较务实

- 默认零配置可跑。
- SQLite 本地可落地。
- 关键路径尽量异步，不阻塞主 Agent。
- 支持 OpenClaw 插件和 Hermes Gateway 两种接法。

### 10.2 记忆结构是可解释的

- 不是黑盒 embedding 池。
- 上层内容有人类可读的 Markdown 载体。
- 能从高层结构回查到底层证据。

### 10.3 对长任务和长周期都分别有方案

- 长任务靠 offload。
- 长周期靠 L0-L3。

很多项目只覆盖其中一边，这个项目两边都做了。

## 11. 这个做法的代价和边界

### 11.1 系统复杂度明显更高

相比简单向量检索，这套方案多了：

- 多层数据模型。
- 后台流水线调度。
- 场景文件维护。
- Persona 增量更新。
- 短期 offload 子系统。

好处是能力更强，代价是维护成本也更高。

### 11.2 质量仍然依赖 LLM 抽取质量

虽然系统做了分层、去重、调度和回钻，但：

- L1 抽取是否准，决定了长期记忆的基础质量。
- L2 场景组织是否好，决定了高层结构是否稳定。
- L3 Persona 是否克制，决定了注入是否有帮助。

所以它不是完全规则驱动，而是“工程框架 + LLM 认知加工”的组合。

### 11.3 上层文件是优势，也是约束

Markdown 让它更白盒，但也意味着：

- 结构设计要谨慎。
- 文件数量增长后需要更强的治理。
- 需要 backup、index、checkpoint 等配套机制维持一致性。

项目里也确实为此配了 backup、scene index、checkpoint、manifest 等基础设施。

## 12. 一句话结论

这个项目的核心做法不是“把历史存起来”，而是把 Agent 记忆拆成两类问题分别处理：

- 对跨会话经验，用 `L0 -> L1 -> L2 -> L3` 做分层沉淀。
- 对单任务上下文膨胀，用 offload + Mermaid 做符号化压缩。

它的本质是一套“可追溯、可解释、可调度、可跨宿主复用”的 Agent Memory 架构，而不是一个单纯的向量检索插件。
