# 多 Agent 任务协作：从教学实现到生产实践

## 一、这个项目怎么实现多 Agent

整体架构分四层，逐步叠加：

```
┌────────────────────────────────────────────────────┐
│ s12: Worktree 隔离 — 每个任务在独立目录执行，互不干扰  │
├────────────────────────────────────────────────────┤
│ s11: 自主认领 — teammate 空闲时轮询任务板，自己找活干   │
├────────────────────────────────────────────────────┤
│ s10: 协议层 — shutdown/plan approval 请求-响应状态机  │
├────────────────────────────────────────────────────┤
│ s09: 团队通信 — 持久 teammate + JSONL 邮箱           │
├────────────────────────────────────────────────────┤
│ s07: 任务系统 — 文件持久化的 task graph + 依赖关系     │
├────────────────────────────────────────────────────┤
│ s04: Subagent — 临时子agent，独立上下文，用完即弃      │
├────────────────────────────────────────────────────┤
│ s01: Agent Loop — while + tool_use 基础循环           │
└────────────────────────────────────────────────────┘
```

### 核心机制

#### 1. 任务板（控制面）

```json
// .tasks/task_12.json
{"id": 12, "subject": "重构auth模块", "status": "pending", "assignee": null, "depends_on": [11]}
```

所有工作拆成有依赖关系的 task 文件，持久化在磁盘。这是协调的骨架。

#### 2. JSONL 邮箱（通信面）

```
.team/inbox/lead.jsonl    — 每人一个文件
.team/inbox/alice.jsonl   — 追加写入，读后清空
```

- 发消息 = 往目标文件追加一行 JSON
- 收消息 = 读取自己的文件全部内容，然后清空
- 每轮 agent loop 开头先检查收件箱

邮箱消息格式：

```json
{"type": "message", "from": "lead", "content": "请修复auth模块的bug", "timestamp": 1715742000.0}
```

5 种消息类型：

| 类型 | 用途 | 引入阶段 |
|------|------|----------|
| `message` | 普通点对点通信 | s09 |
| `broadcast` | 群发所有 teammate | s09 |
| `shutdown_request` | 请求某个 teammate 关闭 | s10 |
| `shutdown_response` | 回复同意/拒绝关闭 | s10 |
| `plan_approval_response` | 回复同意/拒绝计划 | s10 |

核心实现：

```python
class MessageBus:
    def send(self, sender, to, content, msg_type="message"):
        """写一行JSON到目标的收件箱文件"""
        msg = {"type": msg_type, "from": sender, "content": content, "timestamp": time.time()}
        with open(f"inbox/{to}.jsonl", "a") as f:
            f.write(json.dumps(msg) + "\n")

    def read_inbox(self, name):
        """读取全部消息，然后清空文件（drain模式）"""
        msgs = [json.loads(line) for line in open(f"inbox/{name}.jsonl")]
        open(f"inbox/{name}.jsonl", "w").close()
        return msgs

    def broadcast(self, sender, content, teammates):
        """给除自己外的所有人各发一条"""
        for name in teammates:
            if name != sender:
                self.send(sender, name, content, "broadcast")
```

#### 3. 自主认领（调度面）

```python
# teammate 空闲时每5秒轮询
scan .tasks/ → 找到 status=pending & assignee=null → 认领 → 开始工作
```

没有中央调度器分配任务，teammate 自己找活。

#### 4. Worktree 隔离（执行面）

每个 task 绑定一个 git worktree 目录，teammate 在各自目录里工作，文件系统级别互不干扰。

```json
// .worktrees/index.json
{
  "worktrees": [{
    "name": "auth-refactor",
    "path": ".../.worktrees/auth-refactor",
    "branch": "wt/auth-refactor",
    "task_id": 12,
    "status": "active"
  }]
}
```

## 二、三种通信机制如何配合

```
┌─────────────────────────────────────────────────────────┐
│  Lead Agent                                             │
│                                                         │
│  1. 拆解大目标 → 写入共享任务板 (s07)                      │
│  2. 简单子任务 → 直接 spawn subagent (s04, 同步等结果)     │
│  3. 复杂/并行任务 → 委派给 teammates (s09, 异步通信)       │
└─────────────────────────────────────────────────────────┘
         │                          │
    spawn subagent             spawn teammate
         │                          │
         ▼                          ▼
┌─────────────────┐     ┌─────────────────────────┐
│ Subagent (临时)  │     │ Teammate (持久)          │
│ 独立messages[]   │     │ 独立线程 + 独立messages[] │
│ 做完返回，销毁   │     │ 通过邮箱异步通信          │
└─────────────────┘     │ 空闲时自主认领任务        │
                        └─────────────────────────┘
```

| 机制 | 场景 | 方式 |
|------|------|------|
| 独立 messages[] | 父→子委派轻量任务 | 同步调用，结果返回 |
| JSONL 邮箱 | 平级 teammate 间协作 | 异步文件追加写 |
| 共享任务板 | 自主协作，无需分配 | 扫描 + 认领 |

### 典型协作流程

```
1. 用户: "重构这个项目的认证模块"

2. Lead agent:
   - 拆成5个task写入任务板，标注依赖关系

3. Teammate-A 空闲扫描 → 认领 task-1 "重写auth中间件"
   Teammate-B 空闲扫描 → 认领 task-2 "更新登录页面"
   （自主认领，并行执行）

4. Teammate-B 需要新的API接口格式 →
   发邮箱消息给 Teammate-A
   Teammate-A 回复接口定义（请求-响应协议）

5. Lead agent 需要快速确认某个文件内容 →
   spawn subagent 读文件返回摘要（同步子调用）

6. 各teammate完成后更新任务板状态
   Lead agent 看到所有task done → 汇总报告给用户
```

## 三、生产级 Multi-Agent 是这样吗？

**核心模式一致，工程细节差异巨大。**

| 维度 | 教学实现（本项目） | 生产级实现 |
|------|-------------------|-----------|
| **通信** | 本地 JSONL 文件追加 | 消息队列（Redis Stream / Kafka / NATS）、gRPC、WebSocket |
| **并发** | Python threading，无锁 | 进程隔离 / 容器隔离 / 分布式节点，带锁和事务 |
| **任务调度** | 文件轮询，先到先得 | 优先级队列、能力匹配、负载均衡、重试/超时/死信 |
| **状态管理** | JSON 文件 | 数据库（Postgres / Redis），支持崩溃恢复 |
| **隔离** | git worktree | Docker 容器 / VM / 沙箱（如 E2B、Firecracker） |
| **可观测性** | 打印日志 | 结构化 tracing（OpenTelemetry）、metrics、dashboard |
| **容错** | 无（崩了就没了） | 心跳检测、自动重启、任务重新分配、幂等执行 |
| **权限** | 简单路径检查 | RBAC、沙箱策略、审批工作流 |

### 生产级典型架构

```
                    ┌─────────────┐
                    │ Orchestrator│  (可能也是LLM，也可能是规则引擎)
                    └──────┬──────┘
                           │ 任务分发
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Worker 1 │ │ Worker 2 │ │ Worker 3 │  (各自容器/进程)
        │ (LLM)    │ │ (LLM)    │ │ (LLM)    │
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │             │             │
             └─────────────┼─────────────┘
                           ▼
                  消息总线 / 共享状态存储
```

### 当前主流生产方案

**Claude Code（Anthropic）**— 本项目的原型：
- Subagent 通过 fork 进程实现，不是线程
- 通信走内存管道，不是文件
- 但核心模式相同：独立上下文 + 结果返回

**OpenAI Codex / Devin / Factory 类产品：**
- 每个 agent 跑在独立沙箱容器里
- 通过 API/事件总线通信
- 有专门的 orchestrator 层做任务拆分和分配

**CrewAI / AutoGen / LangGraph（框架层）：**
- 定义 agent 角色 + 通信拓扑
- 底层用内存队列或函数调用传递消息
- 本质和本项目的邮箱模式相同，只是封装更厚

## 四、本项目的教学价值

这个项目剥掉了所有工程复杂度，**只保留了多 agent 协作的本质模式**：

1. **独立上下文** — 每个 agent 有自己的 messages[]，互不污染
2. **异步通信** — 通过共享介质（文件/队列）交换消息
3. **共享任务状态** — 通过持久化的任务板协调谁做什么
4. **自主决策** — 模型自己决定何时发消息、认领什么任务、如何协作

把 JSONL 换成 Redis，把线程换成容器，把文件轮询换成事件订阅——**模式不变，只是基础设施升级**。

### 生产级的复杂度不在架构，在工程

- 一个 agent 挂了怎么办 → 容错
- 两个 agent 同时认领同一个任务怎么办 → 并发控制
- 任务做了一半 token 用完怎么办 → 上下文管理
- 怎么知道哪个 agent 在摸鱼 → 可观测性
- 怎么防止 agent 搞破坏 → 权限沙箱

这些是工程问题，不是架构问题。架构就是本项目展示的这些。

---

## 五、一句话总结

> **多 Agent = 独立循环 + 异步通信 + 共享状态 + 自主决策。** 教学用文件实现，生产用基础设施实现，但模式完全相同。
