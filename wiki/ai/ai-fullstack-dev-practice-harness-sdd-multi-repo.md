# Harness + SDD + 多仓管理：AI 全栈开发实践

**来源：** [得物技术, 2026-05-06](https://mp.weixin.qq.com/)

## 核心结论

这篇文章的核心不是“让 AI 帮你写代码”，而是给出一套让 AI 更稳定地产出团队可采纳代码的工程约束体系。它把可用实践归结为三件事：

- **Harness 思维**：不要让 AI 凭空创造，而要让它模仿现有实现
- **SDD 驱动**：先把前后端需求和接口契约拆清楚，再让 agent 分头执行
- **多仓统一工作区 + 多 Agent 并行**：让 AI 同时看见前后端上下文，并把独立任务并行化

## Harness 思维

文章把“AI 最容易产出外星代码”定义为全栈 AI 开发的首要问题。它给出的解法不是更换模型，而是增强约束：

- 找到功能最相似的已有实现作为参照
- 优先复用现有组件、接口封装、数据结构
- 明确参考文件、参考接口和命名风格
- 宁可“照着抄一份再改”，也不要让 AI 自由发挥

这和“prompt 越长越好”不是一回事，重点在于**约束生成范围**，而不是泛泛描述需求。

## 为什么要多仓同工作区

文章强调，前后端分仓但放进同一工作区，有三个直接收益：

- AI 能同时看到前端调用方式和后端返回结构，减少字段对不上的问题
- 代码索引可以跨仓语义检索，自动找到相似实现链路
- 前后端的 SDD 文档可以集中管理，便于接口契约对齐

它对 Cursor 和 Claude Code 的对比也很实用：Cursor 更偏快速迭代和索引检索；Claude Code 适合长链路复杂任务，但多 Agent 与文件引用体验更依赖手工组织。

## SDD 的作用

文中把 SDD 视为“先把问题说清，再让 agent 执行”的中间层。对全栈场景，关键要求有三点：

- 前后端各自生成一份 SDD
- 接口契约必须对齐
- 字段映射必须在前后端文档里明确对应

值得注意的是，作者特别提醒：**SDD 不等于需求文档**。它真正有价值的地方，是把隐含设计问题提前暴露出来，例如：

- 主键怎么设计
- 排序接口如何批量更新
- 嵌套对象是拆表还是 JSON 存储
- 哪些字段应该由后端自动生成而不是前端传入

## 多 Agent 协作

文章认为，SDD 生成后，前后端代码生成天然适合并行，因此多 Agent 的价值不在“更聪明”，而在：

- 让前端和后端任务同时推进
- 保持各自上下文专注，减少串扰
- 把联调放到后面，而不是一开始就混在一个长对话里

## 验证与风险

文章最后一部分最有价值：它没有把 SDD 神化，而是强调三阶段验证、Mock 数据、后端独立构建验证，以及测试尽早介入。核心提醒是：

- SDD 不能替代测试
- 隐性功能点如果不显式写出，AI 很容易漏掉
- 真正的效率提升来自“约束 + 验证”，不是“把更多代码交给 AI”
