# 评测 Agent 的统一自动化架构

**来源：** [翠翰, 2026-02-05](https://mp.weixin.qq.com/s/5ytev_RLKutH9S6mgmMQpw)

## 核心结论

这篇文章最值得保留的地方，是它把“评测”从一个个分散的打分任务，重新组织成了统一 agent 架构：让 agent 先学习业务标注标准，再覆盖评测集生成、自动打分、验收和 badcase 分析整条链路。

它真正解决的问题不是“某个指标能不能自动打”，而是当业务场景和评测任务数量快速膨胀时，如何避免评测能力仍然停留在一个个定制脚本和局部 task-head 的拼装状态。

## 这篇的出发点很现实

文章背景很清楚：

- 场景从单一助理扩展到多个对话式助理和 AIGC 场景
- 评测任务和样本数快速增长
- 现有人力无法为每个细分指标持续开发专门方法

因此作者要解决的是“一个评测同学如何同时应对数十项评测任务”的问题。这个目标决定了方案必须优先追求：

- 标准获取自动化
- 评测流程统一化
- 人工干预最小化

## 最有价值的是“让 agent 学文档”

这篇的核心设计不是把评测规则硬编码到 prompt，而是让评测 agent 先学习业务标注文档和评测标准，再生成摘要和打分 prompt。

这个思路很重要，因为它把：

- 培训评测小二
- 配置自动机审
- 维护评测标准

统一到了同一份文档资产上。

这和 [[wiki/ai/agentscope-skills-progressive-disclosure|AgentScope Skills 与渐进式披露]]、[[wiki/ai/transition-from-traditional-to-llm-programming|从传统编程转向大模型编程]] 的方向很一致：真正可扩展的不是 prompt 本身，而是可被 agent 学习和执行的外部规范。

## 它覆盖的是“评测全链路”，不是单点自动打分

文章强调的四个环节分别是：

- 评测集生成
- 自动打分
- 人审预验收
- badcase 分析

这意味着它不是传统意义上的“打分模型”，而更像一个评测 workflow agent。

特别是“人审预验收”这一步很关键：它不是直接取代人工，而是把人工工作从逐条打标，挪到审阅报告和处理低置信度样本上。

这个设计比“追求 100% 全自动”更现实。

## 识图-推理解耦是这篇最硬的技术点

文章里最值得单独记住的技术方案，是用小尺寸 VLM 先做忠实图像梗概，再把梗概和原图一起交给主模型推理。

核心逻辑是：

- 多模态大模型一旦进入长链推理，视觉幻觉会显著增加
- 与其让同一个模型同时负责“看清”和“想清”
- 不如把“识图”和“推理”拆开

这个方案的优点很实际：

- 架构简单
- 成本低
- 对现有基座模型改造小
- 能明显降低 hallucination

这和 [[wiki/ai/production-prompt-ab-evaluation-engineering-practice|生产级 Prompt 自动化推理评估]] 很互补。那篇主要靠规则工程和 bad case 迭代收敛判断，这篇更进一步，在评测基础设施层面处理多模态输入质量。

## 它对“评测 agent”定义得更像基础设施

文章里另一个重要点是：统一评测 agent 不只是模型调用封装，而是一套带记忆、检索、图像预处理、主模型推理和置信度评估的完整架构。

其中几个关键设计值得记：

- 从任务记忆中检索文档、错题、问卷等上下文
- 在需要时补充常识知识召回
- 对图像按策略分类并生成不同风格梗概
- 最终输出文本结果和概率分数，用于决定哪些样本需要人工复核

这里的置信度评估尤其关键，因为它直接决定自动化覆盖率和人工复核成本之间的平衡。

## 我的判断与保留

我认为这篇最大的价值，不是具体用了哪个模型或训练 recipe，而是它把评测标准、评测流程、模型结构和人工复核机制放在了同一个统一架构里。

它的限制也很明确：

- 很多实验结果是场景内经验，不一定能直接迁移到别的业务
- 训练细节很多，但文章更像工程报告，不是完整可复现实验论文
- 统一 agent 架构并不自动等于统一标准质量，文档本身仍然可能有偏差或不一致

所以更准确地说，这篇展示的是“评测能力如何 agent 化、流程化、基础设施化”，不是证明某个单一模型路线已经通吃所有评测任务。
