# 在智能体优先的世界中利用 Codex

**来源：** [OpenAI Engineering, 2026-05-05](https://openai.com/zh-Hans-CN/index/harness-engineering/)

## 核心结论

这篇文章最重要的观点不是“Codex 能写很多代码”，而是：当代码几乎全部由智能体生成时，工程团队的核心工作会从“写代码”转向“设计环境、约束、反馈回路和记录系统”。

OpenAI 团队用 3 名工程师在约 5 个月内，靠 Codex 从空仓库构建出一个约 100 万行代码、1500 个 PR 的内部 beta 产品，并坚持 **不手写代码** 这一限制。文章给出的结论是：真正稀缺的不是编码能力，而是人类的时间、注意力与判断力。

## 工程师角色的变化

在这种工作流里，人类不再主要负责写实现，而是负责：

- 拆解目标，定义任务边界
- 设计工具、抽象和约束
- 审核结果是否满足真实需求
- 把每次失败转译成可复用的规则、文档或自动化检查

文章明确提出一个口号：**人类掌舵，智能体执行。**

## 为什么要让系统对智能体可读

文中最有价值的一点是把“agent-first”具体化为“仓库、UI、日志、指标、追踪都要能被智能体直接读取和操作”。

他们为 Codex 提供了：

- 基于 git worktree 的独立应用实例
- Chrome DevTools 能力，用于驱动 UI、截图、导航和复现 bug
- 本地可观测性堆栈，用 LogQL / PromQL 查询日志和指标

结果是，Codex 不只是改代码，而是能：

- 复现故障
- 观察运行时信号
- 施加修复
- 重启应用
- 再次验证，直到通过

## AGENTS.md 不该是百科全书

文章非常明确地反对“大而全的 `AGENTS.md`”。原因有四个：

- 情境窗口稀缺，巨型说明会挤掉真正和任务相关的信息
- 指导过多时，智能体会做表层模式匹配，而不是有意识导航
- 长文档极易腐烂，变成过时规则坟场
- 单个大 blob 难以机械验证覆盖率、新鲜度和交叉链接

他们的替代方案是：

- 用短 `AGENTS.md` 当“目录”和入口地图
- 把真实知识沉淀到结构化 `docs/` 目录
- 用 lint、CI 和后台 doc-gardening agent 持续维护文档质量

这和你当前把 `AGENTS.md` / `CLAUDE.md` 当 schema、并逐步引入 `ingest-state.yaml` 的方向高度一致。

## 代码仓库是记录系统

文章反复强调：对智能体来说，**不在仓库里的知识就等于不存在**。Google Docs、Slack、脑中共识都不可见。可见的只有版本控制下的代码、Markdown、schema、执行计划和可读工件。

因此他们把这些内容都推入仓库：

- 架构地图
- 产品规格
- 执行计划与决策日志
- 质量评分
- 安全与可靠性约束

这套设计的目的不是“多写文档”，而是让智能体能渐进式获取正确信息，而不是被一次性灌输。

## 约束比品味更重要

文章把架构约束视为智能体时代的前置条件，而不是大团队后期才需要的治理：

- 严格的层次边界
- 依赖方向检查
- 自定义 lint 和结构测试
- 小而稳定的“品味不变式”

关键点是：**通过强制不变量，而不是微观规定每一段实现。**  
这样既能保持速度，也能避免架构漂移。

## 吞吐量改变了 merge 哲学

当智能体吞吐量远高于人工注意力时，等待成本开始高于纠错成本。文章因此主张：

- Pull Request 生命周期尽量短
- 偶发失败优先快速重跑，而不是长期阻塞
- 更强调持续修正，而不是一次合并前就追求完美

这套思路只有在自动化验证与回滚能力足够强时才成立。

## 垃圾回收与持续清理

完全自主的智能体会复制仓库中已有的坏模式，因此“AI 残渣”会不断累积。OpenAI 的应对方式不是靠人定期手工扫地，而是：

- 把黄金原则编码成可机械执行规则
- 持续运行后台 Codex 任务扫描偏差
- 自动发起小型重构 PR

文章把它类比为垃圾回收：技术债像高息贷款，持续小额偿还，比放大后一次性处理更可行。
