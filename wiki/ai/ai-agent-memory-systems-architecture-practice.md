# AI Agent 记忆系统

**来源：** [柳遵飞（翼严）, 2026-01-30](https://mp.weixin.qq.com/s/mftM6jr0YiFxRATeNvm5Qg)

## 核心结论

这篇文章的价值在于，它把很多分散讨论的概念重新整理成一层比较清楚的 agent 基础设施图：`短期记忆` 处理单会话上下文，`长期记忆` 处理跨会话个性化和经验沉淀，而 `上下文工程` 和 `记忆服务` 则是让两者可控、可扩展、可集成的运行机制。

它最重要的提醒是：`memory` 不是简单地“多存一点聊天记录”，而是一个和推理、存储、检索、更新策略、安全边界都强绑定的系统设计问题。

## 这篇怎么划分记忆

作者把记忆分成两个核心层次：

- `会话级记忆`：同一 session 内的多轮消息、工具调用和结果
- `跨会话记忆`：从多个 session 中抽取出来、可复用的偏好、事实和经验

这个划分比“短期 / 长期”更准确，因为关键不只是时间长短，而是信息是否能跨 session 复用。

文章还强调二者是双向关系：

- 长期记忆从短期记忆里抽取有效信息
- 长期记忆又会在新会话开始时回灌到短期记忆里辅助推理

所以长期记忆不是聊天记录归档，而是一个不断提炼、注入、更新的循环系统。

## 短期记忆真正对应的是上下文工程

这篇对短期记忆最有价值的一点，是明确把它和上下文工程绑在一起看。

作者总结了三种主要策略：

- `上下文缩减`：保留预览或做摘要，直接减少 token
- `上下文卸载`：把完整内容移到外部存储，只在上下文里留引用
- `上下文隔离`：把任务拆给子 agent，各自维护更小上下文

这个分类很实用，因为它把常见手段按“是否丢信息”和“是否迁出会话”区分开了。

和 [[wiki/ai/react-to-ralph-loop-agent-iteration|从 ReAct 到 Ralph Loop]] 放在一起看，会更容易理解：`Ralph Loop` 解决的是“如何继续工作直到完成”，而这篇讨论的是“工作持续进行时，哪些状态留在上下文里，哪些状态要压缩、卸载或外移”。

## 长期记忆更像个独立子系统

文章认为长期记忆在工程上通常会独立成第三方组件，这个判断是对的。它给出的通用组件栈包括：

- `LLM`：从短期记忆中抽取事实、偏好、经验
- `Embedder`：将信息向量化
- `VectorStore`：做语义检索
- `GraphStore`：补复杂关系
- `Reranker`：提高召回结果相关性
- `SQLite` 等操作日志：保留审计与回溯能力

这意味着“长期记忆”本质上更接近一套面向 agent 的个性化知识层，而不是传统对话历史存储。

## 这篇对 RAG 和 Memory 的区分有帮助

文章指出，长期记忆和 `RAG` 在技术上都可能使用 embedding、向量库和检索增强，但两者目标不同：

- `RAG` 主要补通用外部知识
- `Memory` 更强调用户相关、任务相关、可更新的个体化信息

这个区分很重要。否则很容易把“知识库检索”误当成“记忆系统”，但实际上后者还需要处理提炼、更新、遗忘、隐私和画像建模。

## 最值得保留的是它把 Memory 当成基础设施

文章后半段提出 `Memory-as-a-Service`，这个判断很值得记下来。

它隐含的意思是：

- 记忆会从某个 agent 的内部功能，变成多 agent 共用的底层能力
- 记忆层会像数据库、缓存、日志系统一样，成为标准化可接入组件
- 真正难点不只是“存起来”，而是如何做生命周期管理、权限控制、多模态支持和检索质量治理

这和 [[wiki/ai/harness-engineering-codex-agent-first|在智能体优先的世界中利用 Codex]] 里“仓库即记录系统”的思路很互补。前者更偏工程仓库内的显式记录，后者更偏用户与 agent 交互层的动态个性化记忆。

## 我的判断与保留

这篇的优点是结构清晰，适合作为记忆系统总览；但它也偏综述式，很多框架和产品名被列出后没有展开对比，技术难点更多是提出问题，而不是给出实证答案。

我认为它真正值得留下的不是某个具体框架用法，而是这几个框架性的判断：

- 记忆要按 session 内外分层
- 短期记忆问题本质上是上下文工程问题
- 长期记忆问题本质上是提取、检索、更新和治理问题
- 记忆层很可能会基础设施化，而不是永远藏在单个 agent 应用内部
