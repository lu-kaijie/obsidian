# 生产级 Prompt 自动化推理评估

**来源：** [张超（码影）, 2026-02-02](https://mp.weixin.qq.com/s/y2Sq_KEFRlDEnFgDsEKxNQ)

## 核心结论

这篇文章真正有价值的地方，不是“用 LLM 看 A/B 实验数据”这个点子本身，而是它把实验下线判断拆成了一套可运行、可解释、可迭代的生产决策系统：业务规则先显式排序，bad case 再反向修正 prompt，最后把人工经验收敛成带排除条件的稳定决策树。

作者做的不是泛泛“让模型帮忙分析”，而是让模型在一套硬约束下，去执行接近人工资深分析师的判定流程。

## 问题定义得很清楚

文章处理的是一种很实际的场景：

- 每天新增大量策略实验
- 人工巡检要花 4-6 小时
- 简单规则引擎太僵硬
- 只看 `p-value` 或单一收益指标又容易误判

尤其在“打包留白实验”里，最难的是识别那些并非一路明显变坏、而是看起来偶有反弹、实则拖累整体收益的灰区策略。

作者给出的结果也说明系统确实有业务价值：

- 人工巡检从 6 小时降到 30 分钟
- 下线策略数显著减少
- 打包实验指标从负向波动转成稳定正向趋势

## 这篇最重要的是规则工程，不是 prompt 文案工程

文章展示的 prompt 很长，但重点不在“写得长”，而在于它把决策逻辑工程化了：

- 明确优先级顺序
- 每级命中后短路退出
- 在主规则之间插入全局收益检查
- 对易误杀场景加排除条件
- 对数据不足场景做保守处理

换句话说，这更像一个“用自然语言承载的决策程序”，而不是普通提示词。

## 规则设计里最值得保留的点

### 1. 它用 bad case 驱动规则收敛

文章最成熟的地方，是每条规则后面都在回答一个问题：

“为什么这里不能简单判断？”

比如：

- 连续负向时，要排除“末日虽仍负但正在快速好转”的情况
- 趋势恶化时，要排除“前负后正”的真实反弹
- 正向衰减时，要排除“虽然衰减，但累计收益仍明显为正”的情况

这说明作者不是把人类经验直接硬编码进去，而是在用误判样本不断修 prompt 的边界。

### 2. 它把“统计判断”改写成“业务决策判断”

文中一个很重要的判断是：`isSignificant` 只能辅助参考，不能作为主要决策依据。

真正决定下线的，是几个更贴近业务的问题：

- 最近几天是在持续恶化还是正常波动
- 累积收益是正还是负
- 当前损失是否在加速扩大
- 反弹是真的趋势逆转，还是噪音

这比纯统计阈值更贴近生产决策。

### 3. 它很强调归一化与可解释输出

作者不是直接看相对变化率，而是明确在流量比例偏差较大时使用归一化后的千人 DAU 变化量。这一点很关键，因为很多 A/B 场景最大的坑不是“模型不会看图”，而是输入口径本身就不统一。

同时输出被限制为严格 `JSON`，这意味着它不是一个给人看的分析助手，而是一个要接入下游流程的工程组件。

## 我的判断与保留

我认为这篇最值得记住的，不是具体阈值本身，而是它的方法：

- 先把人工判断路径显式化
- 再做优先级排序
- 再从 bad case 补排除条件
- 最后用结构化输出接入生产流程

它的问题在于，文章给出的结果还是单场景、短周期试运行经验，离“通用可靠”还有距离。尤其是 68% 准确率说明它已实用，但远没到可以无监督全自动上线的程度。

所以更准确的定位是：它展示了 prompt-based 决策系统怎样从 demo 走向 production，而不是证明 LLM 已经取代统计实验框架。
